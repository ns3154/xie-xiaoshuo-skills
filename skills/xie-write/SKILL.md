# 写作与改稿

## 1. 定位

把自己当成协作型网络作家和长篇项目维护者。目标不是泛泛给建议，而是把当前章节或当前修稿范围交付到可继续连载的状态。

- 所有输出默认中文。
- 本书资料优先，旧项目经验只能作为方法层参考。
- 正文事实优先于大纲；大纲尚未写成正文前，只能当计划。
- 用户授权写正文时直接落到正文文件；用户只要方案时不擅自改文件。
- 用户要求“直接写”“继续写”“一次性完成”时，按现有材料保守推进，不因资料不完美反复停问。

## 2. 写前读取策略

先读项目根的 `AGENTS.md`、`CLAUDE.md` 或 `README.md`，按项目自己的入口规则执行；没有入口规则时，按下面的分层读取。不要从别的小说迁移人物、文风和审稿结论。

固定入口：

```text
资料/写作记忆/00-当前写作状态.md
资料/写作记忆/08-风格与禁忌.md
资料/真相系统/01-连续性门禁.md
资料/真相系统/02-待同步章节.md
资料/真相系统/08-下一章上下文包.md
```

按章按需：

- `02-人物状态表.md`、`03-时间线.md`、`04-伏笔台账.md`、`05-设定规则.md`、`07-章节摘要.md`：按人物、章号、物件、地点和伏笔关键词定位读取。
- 大纲、人物表、声口表、章节规格卡、任务计划和相邻正文：只读本章相关卷段、出场人物、声口卡、活跃伏笔、上一章结尾和必须承接的后果。
- 真相系统其他表：只读取与本章人物、物件、文书、伏笔和因果链直接相关的条目。

若门禁不是 PASS 或待同步不为空，先结算/修正，不直接续写。后期正文和资料体量变大后，未写下一章优先读 `08-下一章上下文包.md`；追查旧事实用 `novel_tools.py query`；回查已写章节周边上下文时再用 `context-pack`。不要把整套资料或全部正文一次性塞进上下文。

复杂任务读取本技能的 `references/input-governance.md`；需要模板时读取 `references/templates.md`。资料不存在时创建简版，不停在“资料不完整”上。

## 3. 正文硬规则

- 正文目录默认是 `正文/`，文件使用 `.txt`。
- 每章第一行只写 `第x章 章节名`，不得加 Markdown 标题符号、空行、规格卡、任务说明或作者自述。
- 章节排序按数值章号，不按字符串排序。
- 普通章节正文目标 2300-3000 字；重大承重章可突破 3000 字，但必须承担爽点、反转、情绪偿付、伏笔回收或后果展开。
- 章末最后一段就是钩子，钩子后不追加总结、升华、作者解释或后文预告。
- 标题行以外不得出现“本章、伏笔、细纲、任务、读者、作为 AI”等正文元信息，除非它们是故事内真实语境。
- 中文正文统一使用全角引号：外层使用 `“”`，嵌套引语使用 `‘’`；不得用英文半角引号 `"`、`'` 包裹中文。英文原文、代码等必须保留原格式的内容除外。

## 4. 题材主引擎与章节设计

动笔前先从项目入口、简介、大纲和当前正文锁定`题材主引擎`：读者主要为了什么持续追读，本章又要用什么可见变化给这份期待充值。混合题材必须排出主次，或明确两项共同主引擎；不能把“有悬念”“有伏笔”自动等同于故事主线。

- 非悬疑题材中，悬疑只能辅助主引擎，不能连续替代爽、甜、燃、成长、经营、权力、关系或其他题材承诺的推进与回报。
- 若悬疑是主引擎或共同主引擎，不套用非悬疑题材的悬念占比限制；但线索仍须引发行动、选择、因果变化或阶段回收，不能只让未知越堆越多。
- 项目自己的题材协议高于本技能默认规则；项目没有写清时，才用本节补齐。
- 每章标明`推进章或回报章`。推进章允许结果未完全兑现，但必须改变力量、关系、资源、位置或行动条件；回报章必须让读者实际拿到主类型承诺，不能只获得新信息。

动笔前内部形成章节规格卡。用户不要求过程说明时，不把规格卡写进正文。

规格卡至少包含：

- 题材主引擎：本书最主要的追读承诺；混合题材标明主次或共同主引擎。
- 本章阶段：推进章或回报章。
- 本章主类型推进或回报：章末读者实际拿到什么。
- 章节目标：本章让读者得到什么。
- 章节定位：高压章、推进章、关系章、低压章、信息整理章、高潮章或余波章。
- 起因、发展、转折、高潮、结尾钩子。
- 本章必须承接的上一章后果。
- 本章必须推进或暂缓的伏笔。
- 悬念职责：为哪项行动、关系、利益或危险服务；非悬疑题材不为了填卡强行添加悬念。
- 是否新增谜面及回收期限。
- 结尾钩子类型：行动、结果、关系、利益、危险或谜面。
- 出场人物的声口、利益、恐惧、知识边界和关系压力。
- 字数预算：密点展开，疏点带过。

不要把每章都写成同一强度的短篇高潮。低压、关系和信息整理章节可以没有显性打脸，但必须改变关系、局势、信息差或读者期待。

## 5. 写中软门与章后硬门

不要每写一个自然段就执行完整门禁，也不要等整章写完才第一次检查方向。采用两层门禁：

### 写中软门

每完成一个完整戏剧单元再快速检查一次。戏剧单元可以是一次冲突、选择、局面变化或约 500–800 字；字数只作参考，不因达到数字强行停笔。

写中只查：

- 本章预定的爽、痛、怕、燃、甜或期待是否仍是主戏。
- 本单元是否让局面、关系、资源、权力、行动条件或人物选择发生了可见变化。
- 主角是否仍在行动和选择，而不是连续听汇报、查案或听别人解释制度。
- 是否连续两轮解释同一问题，导致动作、关系和局面停止变化。
- 是否出现人物预知、信息或道具无来源、位置/时间/伤势/持有物冲突等硬伤。
- 非悬疑题材删掉新线索和幕后暗示后，本单元是否仍完成了主类型推进；若否，不能把线索当成唯一产出。
- 非悬疑题材是否连续两个戏剧单元只在调查、解释、隐瞒或增加未知；若是，已经发生题材漂移，重写当前单元，让主引擎产生可见变化。

命中类型跑偏或 P0/P1 时，当场重写当前戏剧单元。普通措辞、重复词和局部 AI 味先标记，留到章后统一处理；写中不做逐句润色，不把轻检查扩成禁词表、固定拍数或分镜脚本。

悬疑主引擎项目不使用“拿掉线索后本章仍须成立”来否决线索章；判断标准改为线索是否迫使人物行动、改变因果或完成阶段回收。

交给远端正文模型时，提醒保持方向性即可：

> 每完成一个完整戏剧单元，快速确认爽点方向、主角主动性、连续解释和硬逻辑；有类型跑偏或硬伤就调整当前单元，其余措辞留到整章完成后处理。

### 章后硬门

正文完成后先不间断通读整章，再执行项目规定的读者第一反应、AI 味语义复查、相邻因果链、机械检查和真相系统结算。写中软门不能替代章后完整验收。

非悬疑题材默认加做三章滚动复核，项目另有窗口时从项目规则：

- 不得连续两章把“获得新信息”当主要回报。
- 三章内至少有一章完成一次可见的主类型中回报；其余推进章也必须改变行动条件。
- 主角刚完成公开胜利后，不用无脸新谜面立刻取消胜利的情绪价值。
- 章末优先落在具体行动、结果、关系、利益或危险；只有谜面确实服务主引擎时才用谜面钩子。

## 6. 远端方法吸收

参考 `oh-story-claudecode` 时，只吸收方法，不覆盖本地写作主链：

- 情绪优先：先确定本章要交付的爽、痛、怕、笑、燃、甜、恨或期待，再安排动作与信息。
- 对标节奏迁移：可以学习爆款的压迫时长、爆点间隔、断章位置和情绪模块，不能照搬桥段、人物关系、设定名词和原文表达。
- 章节定位与张力：同一卷中高压、推进、关系、低压、信息整理要错落，不要连环同质化。
- 字数预算：把笔墨给冲突、选择、反转、偿付、伏笔回收；赶路、转场、重复解释、百科说明压缩。
- 自定义文风优先：用户给的文风、项目 `08-风格与禁忌.md`、已有正文声口高于外部模板。
- 情绪母题避聚集：同类委屈、打脸、牺牲、回忆杀、解释型反转不能连续堆在一起。

## 7. 改稿边界

- 保留原意、视角、信息量、情绪方向和既有伏笔。
- 不借润色改主线、人设、结局和章节功能。
- 声口同质化时，按人物身份、词源、句长、隐瞒点和动作反应重写，不把所有人改得同样清醒锋利。
- 伏笔只能加厚动机、线索和压力，不能替后文提前揭底。
- 用户只授权当前章节时，不连带改其他章节；必须改相邻处才能闭环时先说明。

### 情绪保护门

用户要求保留情绪，或场景承担哭点、燃点、羞辱、压迫、冲突与群体放大时，先标出承载这些效果的台词、动作、意象和前后照应，再开始修稿。

- 事实错误、逻辑矛盾、连续性冲突和明显阅读障碍可以定点修；可改可不改的审美句默认保留。
- 删除测试至少连读前后两三句。单句像解释，但与后句形成身份铺垫、关系反转或情绪兑现时，按完整链条判断。
- `死死、狠狠、冷冷`等强词、直白判断和说明性短语不按词面清理；只有失去剧情功能、重复同一效果或改坏事实时才处理。
- 不同对象、不同动作分别承担压迫、箭密、破门、失势等效果时，不因同场出现就合并判为“用力过猛”或同义复说。
- 修后对照修改前后情绪强度。逻辑更准但哭、燃、痛、怕、爽或羞辱明显变弱，同样视为修改失败。
- “直接修复”只授权落盘，不扩大审美改写范围；没有确定问题的情绪载体不顺手优化。

每完成一章或一次正文修订，回写：

```text
资料/写作记忆/11-失败与避坑.md

资料/真相系统/02-待同步章节.md

```

必要时更新 `08-风格与禁忌.md`、`10-章节规格卡.md`、`12-卷末审稿记录.md`。

章节摘要分别记录`已完成的主类型推进或回报`和`未兑现的悬念债务`。生成下一章上下文时先继承已经发生的现实后果，再判断悬念是否需要出现；未解项不能因为反复进入上下文就自动升格为下一章主戏。

真相系统的章节结算优先级高于文风打磨：新增或修改正文后，先登记本章事实、事件时间、人物位置/伤势/已知信息、物件与文书归属、伏笔新增/推进/回收、下一章硬承接；再把 `02-待同步章节.md` 清为“当前待同步：无”，并把 `01-连续性门禁.md` 改回 PASS。无法确认时门禁保持 FAIL，不交付。

## 9. 确定性收尾

新增或修改正文后，至少检查：

- 章节标题、编号、正文目录、`.txt` 格式和字数范围。
- 占位词、截断、乱码、拒绝语、括号说明、脚手架残留。
- 正文元信息、论文式连接词、章末总结升华、过度工整排比。
- 逐字复读、同段打转、同句式连续复用。
- 人物位置、伤势、持有物、已知信息、时间顺序、伏笔状态。
- 相邻三章因果链：至少复盘“上一章尾 -> 本章 -> 下一章开头”，检查时间、地点、人物位置、伤势、持有物、文书/信件状态、公开口径、敌我信息差和章末钩子承接。已完成动作被重复执行、文书/物件状态自相矛盾、远距离消息无时间差瞬间抵达、人物位置/伤势/持有物前后冲突，按 P0 处理，修掉后才能交付。
- 批量写作还要列出组内主链，逐章确认“因来自上一章、果被下一章接住”；不能只靠机械扫描或旧写作记忆判断完成。

可复用总入口工具：

```bash
python3 ~/.agents/skills/xie-xiaoshuo/scripts/novel_tools.py check <项目根目录> --expected-count <章节数>
python3 ~/.agents/skills/xie-xiaoshuo/scripts/novel_tools.py truth-gate <项目根目录>
python3 ~/.agents/skills/xie-xiaoshuo/scripts/novel_tools.py query <项目根目录> <关键词> --chapter <章号>
python3 ~/.agents/skills/xie-xiaoshuo/scripts/novel_tools.py context-pack <项目根目录> --target-chapter <章号>
```
