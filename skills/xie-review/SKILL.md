# 审稿与送审

## 1. 定位

默认进入 review-only。先看清范围和证据，再给结论；除非用户授权，不主动续写、不大改正文、不替作者改结局。

- 审稿开头说明覆盖范围：读了哪些文件、是否有正文、是否完整阅读、哪些结论只是大纲判断。
- 正文未创建前，只能做大纲审查、设定审查、开写前风险审查，不能冒充全文审稿。
- 发现旧项目记忆与本项目资料冲突时，以本项目资料为准。
- 修稿默认最小修补：硬矛盾、章号/回指、脚手架残留、语言噪音、明确点名的问题。

## 2. 审稿流程

1. 锁定范围：章节区间、正文目录、版本、是否允许改文件。
2. 读取项目规则、正文、写作记忆、章节摘要、人物状态、时间线、伏笔台账、真相系统和既有审稿报告。
3. 先做确定性预检，再做人类编辑判断。
4. 建问题池，所有重要问题必须能定位到文件、章节、场景或原文线索。
5. 区分硬伤、平台风险、审美建议和可保留选择。
6. 输出结论、分级、证据、最小修补路线；只有授权后才改正文。

确定性预检包括：

- 章节缺漏、编号乱序、标题格式、字数异常。
- `TODO/TBD/FIXME/待补/占位`。
- 正文元信息、模型退化、拒绝语、重复段、乱码。
- AI 句式、论文式连接词、章末总结升华。
- 人物声口同质化、时间线矛盾、伏笔状态不明。
- 真相系统门禁：`资料/真相系统/` 缺失、`02-待同步章节.md` 未清零、`01-连续性门禁.md` 不是 PASS、`03-事实总账.jsonl` 或 `04-事件时间线.jsonl` 缺少已存在正文章号。
- 相邻三章因果链断裂：至少抽查“上一章尾 -> 本章 -> 下一章开头”的时间、地点、人物位置、伤势、持有物、文书/信件状态、公开口径、敌我信息差和章末钩子承接；卷末/全文审稿按章号连续检查主链。

### 情绪保护门

审读哭点、燃点、羞辱、压迫、冲突和群体回应时，先标出情绪载体及其前后照应，再判断是否需要修改。

- 只有事实错误、逻辑矛盾或明显阅读障碍，才进入直接修复池；单纯“可以更短、更克制”不构成问题。
- 删除测试至少连读前后两三句。前句立身份、关系或压力，后句完成反转、抛弃或偿付时，按整条情绪链判断。
- 强词、直白判断、说明性短语和群体放大不因词面像 AI 就降级；先确认它是否改变人物关系、现场压力或读者回报。
- 不同对象、不同动作分别放大盾阵、箭雨、破门、溃逃等结果时，不因集中在同一高压场景就判成一组模板套语。
- 审美分歧、烈度选择和已有声口列为可保留选择。用户授权直接修复，也不把可保留选择自动升级为正文改动。
- 修后若事实更准但情绪明显变弱，验收仍不通过；最小修补必须同时保住情绪方向和强度。

## 3. 分级口径

使用 `P0/P1/P2/P3`：

- `P0`：会让作品无法成立或无法交付的问题。
- `P1`：强影响理解、追读、人物可信度或主线闭环的问题。
- `P2`：影响阅读质感、节奏、声口、AI 味或局部可信度的问题。
- `P3`：可选优化、包装建议、局部表达打磨。

章节因果链硬矛盾默认按 `P0`：已完成动作被重复执行、文书/物件状态自相矛盾、远距离消息无时间差瞬间抵达、人物位置/伤势/持有物前后冲突、上一章章末钩子被下一章无解释跳过。只有明确是局部措辞不清且不影响剧情成立时，才可降为 P1/P2。

真相系统门禁失败默认按 `P0`：待同步章节不为空、门禁 FAIL、事实总账/事件时间线缺已写章节，意味着下一轮续写缺少确定事实源。只有纯文档文件名迁移、且正文和台账已可互证时，才可降级为 P1 文档维护问题。

参考远端 `story-review` 的多视角口径时，可先用 `S1/S2/S3/S4 findings` 建池，再映射到本地分级：

- `S1` 通常对应 `P0/P1`。
- `S2` 通常对应 `P1/P2`。
- `S3` 通常对应 `P2`。
- `S4` 通常对应 `P3`。

映射必须说明理由，不机械改级。

## 4. 审稿模式声明

深度审稿报告开头固定写：

```markdown
Requested Mode:
Effective Mode:
Fallback:
Rubric:
Rubric Source:
Coverage:
```

当多代理、外部 rubric、参考文件或审稿 agent 不可用时，降级为 solo 审稿，并说明 fallback。不能把部分执行结果包装成完整多视角结论。

内置 fallback rubric：

- 开篇与留存：前三章承诺、题材卖点、主角动机、读者期待。
- 结构与节奏：卷目标、冲突递进、低压章功能、高潮偿付、章末钩子。
- 人物与声口：动机、关系变化、台词词源、权力边界、OOC 风险。
- 逻辑与连续性：时间、空间、物品、伤病、信息差、设定代价。
- 伏笔与回收：种子、推进、延迟理由、回收证据、开放线合理性。
- 文风与 AI 味：机械句式、抽象总结、过度工整、作者观点容器。
- 平台与包装：题材定位、简介卖点、合规风险、编辑第一眼。

## 5. 卷末与全文审稿

卷末审稿必须按章号数值顺序回读本卷全部 `.txt` 正文，并结合本卷写作记忆。不要只凭写作过程记忆下结论。

全文或终审任务默认只处理 `正文/` 下正式章节，其他目录只作辅助。先建章节索引、时间线、人物线、物件/文书状态、伏笔台账、真相系统对账和问题池，再输出结论。

如果用户授权修稿：

- 默认只自动修一轮。
- P0/P1 优先，P2/P3 只修明确影响阅读、声口、AI 味或连续性的点。
- 修后无净提升、证据不足或引入新矛盾时停止，留下未解决问题。

## 6. 完稿与送审

完稿审读包、二轮复审、锁稿、送审包装和项目清理读取 `references/review-and-packaging.md`。

核心边界：

- 第一轮专业 AI 审稿只给正文和中性导航资料；提示词单独粘贴。
- 作者自评、担忧、已知弱点、不得改动项放第二轮对账，不塞进首轮上传包。
- 外部评审回来后逐条落账：接受开放、必须补账、可不改。
- 无 P0/P1 时默认停改正文，重点转包装、简介、梗概、卖点定位。
- 清理项目默认移入 `bak/`，不直接永久删除，不碰 `正文/`。

可复用总入口工具：

```bash
python3 ~/.agents/skills/xie-xiaoshuo/scripts/novel_tools.py stats <项目根目录> --details
python3 ~/.agents/skills/xie-xiaoshuo/scripts/novel_tools.py check <项目根目录> --expected-count <章节数>
python3 ~/.agents/skills/xie-xiaoshuo/scripts/novel_tools.py truth-gate <项目根目录>
python3 ~/.agents/skills/xie-xiaoshuo/scripts/novel_tools.py zip <项目根目录> <输出zip> --include 正文 --include 审稿数据包
```
